CI/CD (Continuous Integration and Continuous Deployment/Delivery) kisi bhi modern software development ka ek sabse important hissa hai. Chahe aapka project chhota (normal) ho ya bada (live/production), CI/CD aapke code likhne se lekar use server par live karne tak ke process ko automate kar deta hai.

Chaliye isko Beginner se lekar Advance level tak samajhte hain.

1. Beginner Level: CI/CD Kya Hai? (The Basics)
Jab bhi multiple developers ek project par kaam karte hain, toh code ko merge karna, test karna, aur server par daalna ek lamba aur risky process hota hai. CI/CD isko aasan banata hai.

"CI (Continuous Integration)": Iska matlab hai ki jaise hi developer apna code repository (jaise GitHub/GitLab) par push karta hai, ek automated system us code ko uthata hai, usko Build karta hai, aur us par Automated Tests run karta hai. Isse ye faida hota hai ki agar naye code ne purane code ko break kiya hai, toh turant pata chal jata hai.

"CD (Continuous Delivery / Deployment)":"Continuous Delivery: CI pass hone ke baad, code deploy hone ke liye ready hota hai, lekin production (live) par bhejne ke liye ek manual click (approval) ki zaroorat hoti hai.

Continuous Deployment: CI pass hone ke baad, code automatically live server par chala jata hai bina kisi manual approval ke.
Aasan shabdon mein: Code Likho ➡️ Git Push karo ➡️ (🤖 Auto Build ➡️ 🤖 Auto Test ➡️ 🤖 Auto Deploy) = CI/CD

Popular Tools:

GitHub Actions (Sabse popular aur aasan)
GitLab CI/CD
Jenkins (Thoda purana aur complex, par badi companies mein use hota hai)
Bitbucket Pipelines
CircleCI

2. Intermediate Level: Ek Normal Project Mein CI/CD Kaise Setup Karein?
Maan lijiye aapka ek simple Node.js ya React ka project GitHub par hai. Hum GitHub Actions ka use karke iska CI/CD banayenge.

Step 1: Workflow file banana Aapko apne project ke root folder mein ek file banani hoti hai: .github/workflows/main.yml

Step 2: YAML likhna (Example - Node.js CI) Is file mein hum batate hain ki kya karna hai:
name: Node.js CI/CD Pipeline

# Kab run karna hai?
on:
  push:
    branches: [ "main" ] # Jab bhi 'main' branch me code push ho

# Kya steps run karne hain?
jobs:
  build-and-test:
    runs-on: ubuntu-latest # Ek virtual computer (server) milega

    steps:
    - name: Checkout Code
      uses: actions/checkout@v3 # Code ko virtual server me download karega

    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18.x'

    - name: Install Dependencies
      run: npm install # Saare packages install karega

    - name: Run Tests
      run: npm test # Aapke tests run karega

    # CD Part (Deployment - Example)
    - name: Deploy to Server
      if: success() # Agar test pass hue tabhi deploy karo
      run: |
        echo "Deploying to production server..."
        # Yahan aap apne server ka ssh command ya deployment script dalte hain
===========================
aise hi aap ye code main branch par push karenge, GitHub automatically ek machine start karega, aapka code test karega aur pass hone par deploy kar dega.
Jaise hi aap ye code main branch par push karenge, GitHub automatically ek machine start karega, aapka code test karega aur pass hone par deploy kar dega.

3. Advanced Level: Live / Enterprise Projects Mein CI/CD Kaise Hota Hai?
Badi companies aur live projects mein CI/CD itna simple nahi hota. Wahan bohot saari layers hoti hain taaki live server par koi error na jaye.

Advanced concept jo live projects mein use hote hain:

Multiple Environments (Dev -> Staging -> Prod):

Code pehle Dev server par deploy hota hai.
Testing team usko test karti hai, fir wo Staging (UAT) par jata hai jo bilkul live server jaisa hota hai.
Sab sahi hone par hi wo Production (Live) par jata hai.
Docker & Containerization:

"It works on my machine" wali problem solve karne ke liye, code ko direct deploy nahi karte. Pehle uski ek Docker Image banti hai (ek box jisme code aur uski saari requirements hoti hain).
CI pipeline code ko test karti hai, Docker image banati hai, aur usko Docker Hub ya AWS ECR par push kar deti hai.
Kubernetes (K8s):

Live projects mein deploy karne ke liye Docker containers ko Kubernetes ke through chalaya jata hai taaki agar traffic badhe toh server apne aap resources manage kar le.
DevSecOps (Security Scanning):

Pipeline ke andar hi security checks hote hain (e.g., SonarQube, Snyk). Agar code mein koi vulnerability (security flaw) hai ya kisi ne password hardcode kar diya hai, toh pipeline fail ho jayegi aur code deploy nahi hoga.
Deployment Strategies (Zero Downtime):

Blue-Green Deployment: Do servers hote hain (Blue aur Green). Live user Blue par hote hain, naya code Green par deploy hota hai. Test hone ke baad traffic Green par switch kar dete hain. Isse site 1 second ke liye bhi band nahi hoti.
Canary Deployment: Naya code sirf 10% users ko dikhaya jata hai. Agar unhe koi error nahi aati, tab baaki 90% ko dikhaya jata hai.
Summary: CI/CD Pipeline ka ek Real-world Flow:
Developer code push karta hai (Git).
GitHub Actions / Jenkins trigger hota hai.
Code ka Linting aur Unit Test run hota hai (CI).
Security Code Scan hota hai.
Pass hone par Docker Image banti hai aur registry mein save hoti hai.
CD tool (jaise ArgoCD, AWS CodeDeploy) nayi image ko Staging server par dalta hai.
QA team approve karti hai.
Code Live (Production) server par deploy ho jata hai (CD).
